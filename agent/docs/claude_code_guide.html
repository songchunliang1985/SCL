01 · 安装 Claude Code

⏱ 01:09

Claude Code 是 Anthropic 官方推出的命令行编程助手,直接运行在你的终端里,能读写代码、执行命令、调用工具。它不是一个 IDE 插件,而是一个 CLI 工具。

安装方式

通过 npm 全局安装:

npm install -g @anthropic-ai/claude-code

安装后直接输入 claude 启动。

系统要求

  • Node.js 18+:macOS/Linux/Windows(WSL 推荐)均可
  • Git:强烈建议,大部分项目级功能依赖它
  • 网络:需要访问 Anthropic API
💡 提示
Windows 用户建议通过 WSL2 + Ubuntu 使用,原生 PowerShell 下部分功能(如 sandbox)受限。

升级与卸载

# 升级到最新版
npm update -g @anthropic-ai/claude-code

# 卸载
npm uninstall -g @anthropic-ai/claude-code

02 · 登录与授权

⏱ 01:43

首次运行 claude 会提示你登录。支持两种账户类型:

账户类型适用人群计费方式
Claude Pro / Max个人订阅用户按月订阅,无按 token 计费
Anthropic Console (API)开发者 / 企业按 token 用量计费

登录流程

  1. 终端输入 claude
  2. 选择 "Claude Pro/Max" 或 "Anthropic Console"
  3. 浏览器自动打开,授权后返回
  4. 凭据自动保存到 ~/.claude/,后续无需重复登录

常用账户命令

/login       # 切换账户 / 重新登录
/logout      # 退出登录
/status      # 查看当前账户状态
ℹ️ 关于第三方模型
如果你想用 DeepSeek、Kimi、GPT 等非 Claude 模型,可通过设置 ANTHROPIC_BASE_URL 和 ANTHROPIC_AUTH_TOKEN 环境变量指向兼容端点,和 Pro 账户互不冲突,在不同终端窗口切换使用。

04 · 三种模式详解(默认 / 自动 / 规划)

⏱ 03:12

Claude Code 有三种运行模式,用 Shift+Tab 循环切换,底部状态栏会显示当前模式。

① 默认模式 (Default)

每次文件编辑、命令执行前都会弹窗询问你是否允许。最安全,最慢。

  • 适合:陌生项目、敏感操作、第一次使用
  • Claude 修改一个文件 → 你按 y 确认 → 才会写入

② 自动模式 (Accept Edits)

自动接受文件编辑,但仍会询问危险操作(如 rm -rf)。

  • 适合:信任 Claude 在当前项目的操作时
  • 底部显示 ⏵⏵ accept edits on

③ 规划模式 (Plan Mode)

Claude 只读不写,先给你一份完整的执行计划,你确认后才动手。

  • 适合:大型改动、跨多文件重构
  • 底部显示 ⏸ plan mode on
示例 · 三种模式的同一任务

任务:把项目里所有 console.log 改成 logger.info

  • 默认模式:每改一个文件都问你"是否允许",改 20 个文件按 20 次回车
  • 自动模式:一路畅通,全改完
  • 规划模式:先列出"将修改这 20 个文件的这 47 处",你确认后一次执行

05 · 执行终端命令 (Bash)

⏱ 06:00

Claude Code 内置 Bash 工具,可以执行任何 shell 命令:装依赖、跑测试、启动服务、部署等。

让 Claude 执行命令

示例
你:运行一下单元测试

Claude:[调用 Bash 工具]
  $ npm test
  > jest
  ✓ 14 tests passed

你自己执行命令(! 前缀)

在输入框里以 ! 开头的内容会直接在 shell 里运行,输出会进入对话上下文,方便 Claude 看到结果。

!git status
!ls -la
!python3 main.py
💡 技巧
当你想让 Claude 基于某个命令的输出继续工作时,用 ! 前缀比让 Claude 自己跑命令更直接,因为你能控制执行时机。

06 · 使用规划模式 (Plan Mode)

⏱ 06:49

规划模式是处理复杂任务的关键。Claude 会先通读代码、梳理依赖、制定分步计划,不写任何文件,直到你批准。

进入方式

  • 连按 Shift+Tab 两次(从默认 → 自动 → 规划)
  • 或启动时加参数:claude --permission-mode plan

工作流程

  1. 你描述需求(越详细越好)
  2. Claude 读相关文件、搜索代码、理解架构
  3. Claude 输出分步执行计划
  4. 你 review 计划 → 批准 / 修改意见
  5. 批准后自动退出规划模式,开始实际执行
示例 · 给项目加 Redis 缓存
你:我想给这个 API 项目加 Redis 缓存,缓存用户信息接口

Claude (Plan Mode):
  [读取 routes/user.py, config.py, requirements.txt...]

  📋 执行计划:
  1. 在 requirements.txt 添加 redis==5.0.1
  2. 在 config.py 增加 REDIS_URL 配置项
  3. 新建 cache/redis_client.py 封装连接
  4. 修改 routes/user.py:
     - 在 get_user() 前加缓存读取
     - 在 update_user() 后清理缓存
  5. 在 .env.example 补充 REDIS_URL 示例
  6. 更新 README 部署说明

  是否批准执行?

07 · 跳过所有权限检测 (--dangerously-skip-permissions)

⏱ 11:06

启动时加 --dangerously-skip-permissions(简称 YOLO 模式),所有权限弹窗都被跳过,Claude 可以任意写文件、跑命令。

claude --dangerously-skip-permissions
⚠️ 危险
顾名思义,这个模式会让 Claude 拥有完全控制权,理论上可以 rm -rf ~、删数据库、提交敏感信息到 git。只在一次性容器、沙箱 VM、或完全信任且有备份的环境使用。

适用场景

  • Docker 容器里跑一次性任务
  • Devcontainer / Codespaces 等隔离环境
  • 批量处理脚本(已经人工确认过任务范围)

更安全的替代方案

用细粒度的 permissions 配置代替一刀切:

// ~/.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm:*)",
      "Bash(git status)",
      "Edit",
      "Read"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(sudo:*)"
    ]
  }
}

08 · 后台任务管理 (Background Tasks)

⏱ 13:53

对于长时间运行的命令(开发服务器、构建、日志监听),Claude 可以把它们放到后台,不阻塞对话。

启动后台任务

你:启动开发服务器,让它在后台跑

Claude:[Bash 工具 run_in_background=true]
  $ npm run dev
  → 已在后台启动,任务 ID: bash_1

查看与管理

  • 查看输出:Claude 用 BashOutput 工具读取后台任务的实时输出
  • 停止任务:KillBash 工具终止
  • 多任务并行:可以同时跑前端 dev + 后端 dev + 数据库
示例 · 前后端联调
你:启动前端和后端开发服务器,然后访问 /api/health 看看是否正常

Claude:
  [后台启动 npm run dev (前端)]
  [后台启动 uvicorn main:app (后端)]
  [curl http://localhost:8000/api/health]
  → {"status":"ok"}
  两个服务都运行正常。

09 · 版本回滚 (Rewind)

⏱ 15:35

当 Claude 写的代码方向不对,或你想回到某个中间状态,可以用 Rewind 一键回滚。

触发方式

在输入框连按两次 Esc,进入 Rewind 面板。

三种回滚粒度

选项作用对话历史文件
Conversation only只回滚对话回退保留当前
Code only只回滚代码保留回退到选定点
Both都回滚回退回退
示例场景

你让 Claude 重构一个模块,它改了 10 个文件。测试后发现方案有问题,想回到重构前。

Esc Esc → 选择"重构前"那个检查点 → 选 "Both" → 一秒回到原状,可以换个思路再来。

💡 和 git 的关系
Rewind 是 Claude Code 的会话级快照,和 git 互补。重大改动前最好先 git commit,双保险。

11 · 安装 MCP Server (以 Figma 为例)

⏱ 21:09

MCP (Model Context Protocol) 是 Anthropic 推出的开放协议,让 Claude 能连接外部工具和数据源——数据库、设计工具、搜索引擎、issue 跟踪系统等。

安装命令

# 以 Figma 官方 MCP 为例
claude mcp add figma -- npx -y @figma/mcp-server

# 列出所有已安装的 MCP
claude mcp list

# 移除
claude mcp remove figma

安装作用域

  • --scope user:用户级,所有项目可用
  • --scope project:项目级,存在 .mcp.json,随仓库共享
  • --scope local:仅当前目录本地(默认)

配置认证

大部分 MCP 需要 API token,通过环境变量传入:

claude mcp add figma --env FIGMA_TOKEN=figd_xxxxx -- npx -y @figma/mcp-server
💡 常见 MCP
Figma、GitHub、Linear、Notion、Slack、Postgres、Puppeteer(浏览器自动化)、Playwright、Brave Search 等。在 https://github.com/modelcontextprotocol/servers 可以找到官方和社区列表。

12 · 恢复历史会话 (Resume)

⏱ 21:39

Claude Code 会自动保存每个项目的会话历史,可以随时回到之前的对话继续工作。

三种恢复方式

# ① 恢复上一次的会话(最常用)
claude --continue
claude -c

# ② 列出所有历史会话,选一个
claude --resume
claude -r

# ③ 直接恢复指定 session ID
claude --resume <session-id>

会话存储位置

每个项目的会话存在 ~/.claude/projects/<sanitized-path>/ 下,按项目目录隔离。

典型工作流
周一下午:
  cd ~/myproject && claude
  → "帮我实现用户登录功能" (写了一半)
  → Ctrl+C Ctrl+C 下班

周二上午:
  cd ~/myproject && claude --continue
  → 对话历史、上下文、已读文件全部恢复
  → "继续完成昨天的任务"

14 · 上下文压缩与清除

⏱ 24:49

长对话会消耗大量上下文窗口。Claude Code 提供两个关键命令:

/compact · 压缩上下文

Claude 把前面的对话总结成一份精简摘要,保留关键决策和已完成任务,丢弃中间细节。压缩后可以继续工作,上下文大幅减少。

/compact              # 自动压缩
/compact 只保留架构决策和未完成的TODO   # 指定保留重点

/clear · 清除上下文

完全清空当前对话,开始一个全新会话(但文件改动仍保留)。用于切换到完全无关的新任务。

自动压缩

当对话接近上下文上限时,Claude Code 会自动触发压缩。你可以在 settings.json 配置触发阈值:

{ "autoCompactWindow": 200000 }
ℹ️ 压缩 vs 清除
压缩:记得前面做了什么,但细节丢了 — 适合同一项目的延续工作。
清除:完全失忆 — 适合切换到新任务,避免干扰。

15 · 项目记忆文件 (CLAUDE.md)

⏱ 26:30

CLAUDE.md 是项目级"说明书",Claude Code 每次启动都会自动读取。适合放项目约定、架构说明、避坑指南。

三种层级

位置范围用途
~/.claude/CLAUDE.md所有项目个人偏好(语言、风格)
<项目>/CLAUDE.md当前项目项目约定,提交到 git
<项目>/CLAUDE.local.md本地覆盖个人本地设置,gitignore

快速编辑

# 项目根目录启动 claude
/init          # 自动生成一个 CLAUDE.md 模板
/memory        # 打开 CLAUDE.md 编辑器
示例 · 一份好的 CLAUDE.md
# MyProject

## 技术栈
- FastAPI + PostgreSQL + Redis
- 前端 Vue 3 + Pinia + Vite

## 代码约定
- 所有 API 路由在 routes/,业务逻辑在 services/
- 数据库操作必须用 async SQLAlchemy,不用同步版本
- 日志一律用 loguru,禁止 print

## 运行与测试
- 开发: `uvicorn main:app --reload`
- 测试: `pytest -xvs`
- 部署: `./scripts/deploy.sh`

## 避坑
- 不要改 migrations/ 下已提交的文件,新增 migration
- Redis key 前缀必须用 `myproj:`,避免和其他服务冲突
💡 最佳实践
CLAUDE.md 要精简——它会塞进每次对话的系统提示里。大而全的文档反而会挤占上下文。把不常用的说明放 docs/,在 CLAUDE.md 里只写指针。

16 · Hook

⏱ 29:46

Hook 是在 Claude Code 生命周期的特定时刻自动触发的 shell 命令。可以理解为"事件钩子"——某个动作发生时自动执行你定义的脚本。

核心事件

事件触发时机典型用途
PreToolUse工具执行前校验、拒绝危险命令
PostToolUse工具执行后自动格式化、跑测试
UserPromptSubmit用户发送消息日志、关键词过滤
SessionStart会话开始注入环境信息
StopClaude 停止回复通知、总结
PreCompact上下文压缩前备份重要信息

配置位置

写在 ~/.claude/settings.json 或 .claude/settings.json:

示例 · 自动格式化写入的 TS 文件
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.file_path' | { read -r f; [[ $f =~ \\.tsx?$ ]] && prettier --write \"$f\"; } 2>/dev/null || true"
      }]
    }]
  }
}
示例 · 所有 bash 命令记录到日志
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.command' >> ~/.claude/bash-history.log"
      }]
    }]
  }
}
⚠️ 注意
Hook 执行在你的 shell 下,有完整权限,要谨慎编写。错误的 Hook 可能导致 Claude 异常——比如 PreToolUse 返回非 0 会阻塞工具调用。

17 · Agent Skill

⏱ 33:16

Skill 是一段"按需加载"的领域知识 + 工作流,用 Markdown 写,放在 ~/.claude/skills/ 或项目 .claude/skills/。

Claude 只有在任务匹配 Skill 描述时才会加载它的完整内容,不占日常对话的上下文。

Skill 文件结构

~/.claude/skills/pdf-extract/
├── SKILL.md          # 入口文件(必需)
├── scripts/
│   └── extract.py    # 辅助脚本
└── examples/
    └── sample.pdf

SKILL.md 格式

---
name: pdf-extract
description: 从 PDF 中提取结构化数据。触发:用户提到 PDF 抽取、pdfplumber、OCR 场景
---

# PDF 提取技能

## 工作流
1. 用 pdfplumber 读取 PDF
2. 如果是扫描件,走 scripts/ocr.py
3. 输出 JSON 到指定路径

## 代码模板
...

触发机制

启动时 Claude Code 只读取所有 Skill 的 name + description(很短)。当你的请求和某个 Skill 描述匹配时,Claude 才会完整加载那个 Skill 的 SKILL.md 和附属文件。

示例 · 自定义 Skill
~/.claude/skills/code-review/SKILL.md:

---
name: code-review
description: 按团队规范对代码进行 review,包括安全、性能、可读性三个维度
---

# Code Review 流程

## 检查项
### 安全
- SQL 注入 / XSS / 任意文件读写
- 硬编码密钥、AK/SK
- 输入校验

### 性能
- N+1 查询
- 同步阻塞调用
- 内存泄漏

### 可读性
- 函数长度 <80 行
- 变量命名 snake_case
- 必要的类型注解

## 输出格式
按 🔴高危 / 🟡中危 / 🟢建议 分组

18 · SubAgent

⏱ 36:00

SubAgent 是独立的子进程 Claude 实例,拥有自己的上下文窗口、工具权限、模型选择。主 Claude 可以把一个任务"委派"给 SubAgent,等待结果回来。

为什么需要 SubAgent

  • 隔离上下文:主对话不被研究过程的冗余信息污染
  • 并行:多个 SubAgent 同时跑,加速
  • 角色化:code-reviewer、test-writer、docs-writer 各司其职
  • 省钱:让 Haiku 干杂活,Opus 留给关键决策

定义一个 SubAgent

放在 ~/.claude/agents/<name>.md:

---
name: test-writer
description: 专门给指定函数写 pytest 单元测试
model: haiku
tools: Read, Write, Edit, Bash
---

你是一个单元测试专家,职责:
1. 读取指定的 Python 函数
2. 识别关键分支和边界
3. 用 pytest 写出覆盖率 ≥ 90% 的测试
4. 保存到 tests/ 目录

调用方式

你:用 test-writer agent 给 normalize_path 函数写测试

主 Claude:[调用 Task 工具,subagent_type=test-writer]
  → 独立子进程启动
  → 读 normalize_path.py
  → 生成 tests/test_normalize_path.py
  → 返回摘要
主 Claude:完成,已生成 15 个测试,覆盖 5 个边界场景。
并行场景
你:同时用 security-reviewer、perf-reviewer、test-reviewer 三个 agent 审查这个 PR

主 Claude:[并行调用 3 个 SubAgent]
  → 三份报告同时回来
  → 合并输出

19 · Skill 与 SubAgent 的区别

⏱ 39:17

初学者容易混淆,两者都是"定制化扩展",但定位完全不同:

维度SkillSubAgent
本质知识文档 + 工作流提示独立运行的 Claude 实例
运行空间主对话上下文里(按需加载)独立进程 + 独立上下文
何时加载任务匹配描述时,动态加载显式调用 Task 工具时启动
典型用途"怎么做 X"的指南"替我去做 X"的代理
上下文影响占用主对话的上下文不影响主对话
并行不能并行(就是一段提示)可以多个并行
模型选择沿用主对话模型可指定 haiku/sonnet/opus

怎么选

  • 如果是"遵循某种规范或流程"(比如代码风格、提交信息格式) → Skill
  • 如果是"独立完成一个子任务后返回结果"(比如做调研、写测试、审代码) → SubAgent
  • 如果需要并行或保护主上下文不被污染 → SubAgent
  • 如果只是想注入一些领域知识供参考 → Skill
对比示例

需求:做代码 review

  • 用 Skill 实现:写一个 code-review Skill,每次你说"review 这段代码"时加载,主 Claude 按照规范 review,结果在当前对话输出。占主上下文。
  • 用 SubAgent 实现:写一个 code-reviewer Agent,主 Claude 调用它作为子任务,子进程独立完成 review,只返回总结。主对话上下文干净。

21 · Headless 模式 · 脚本化与自动化

🚀 进阶

Claude Code 不只是交互式 REPL,它本身就是一个命令行程序,可以接受参数、读取 stdin、输出结构化结果,让你把 Claude 嵌进任何 Unix 管道和自动化脚本。

一次性模式 -p / --print

# 发送一个 prompt,打印结果后退出(不进入交互)
claude -p "用一句话总结 README.md 讲了什么"

# 结合管道:让 Claude 读取任意命令的输出
git log --oneline -20 | claude -p "从这些提交里找出可能引入 bug 的"

# 读文件
cat error.log | claude -p "分析这个错误栈,给出可能的根因"

JSON 输出 · 机器可读

# 以 JSON 格式输出,方便脚本解析
claude -p "列出项目的主要模块" --output-format json

# 流式 JSON(每行一个事件,适合实时处理)
claude -p "..." --output-format stream-json

会话延续

# 在脚本中延续上一次会话
claude -p "继续刚才的重构任务" --continue

# 恢复指定 session
claude -p "总结这次会话的成果" --resume <session-id>
实战 · 自动 commit message 生成器
#!/bin/bash
# 放在 ~/.local/bin/ai-commit
msg=$(git diff --cached | claude -p "基于这个 diff 写一条 Conventional Commits 格式的提交信息,只输出信息本身不要解释")
git commit -m "$msg"
实战 · 批量处理错误日志
for log in logs/error-*.txt; do
  result=$(cat "$log" | claude -p "用一句话总结这个错误" --output-format json)
  echo "$(basename $log): $(echo $result | jq -r .result)"
done > error-summary.txt
💡 关键参数
--max-turns N(限制轮数)、--model(临时换模型)、--permission-mode bypassPermissions(脚本里跳过交互)—— 组合起来可以把 Claude 变成一个"可编程的智能 Unix 工具"。

22 · 自定义斜杠命令 (Slash Commands)

🚀 进阶

内置的 /compact、/init 很好用,但真正强大的是你可以定义自己的斜杠命令——本质是一段带参数的 prompt 模板。

创建命令

在 ~/.claude/commands/(用户级)或 .claude/commands/(项目级)放 Markdown 文件,文件名就是命令名:

~/.claude/commands/release.md  →  /release

命令文件格式

---
description: 发布一个新版本到 npm
argument-hint: <version>
allowed-tools: Bash(npm:*), Bash(git:*), Edit, Read
---

请帮我发布版本 $ARGUMENTS,步骤:
1. 更新 package.json 的 version 字段
2. 运行 npm test
3. git commit -am "chore: release $ARGUMENTS"
4. git tag v$ARGUMENTS
5. npm publish
6. git push && git push --tags

每一步失败都停下来问我。

使用

/release 1.2.3
# → $ARGUMENTS 被替换成 "1.2.3",然后整个 prompt 发给 Claude

动态参数技巧

  • $ARGUMENTS — 用户传入的完整参数字符串
  • $1 $2 $3 — 按空格切分的位置参数
  • 命令内部可以用反引号执行 shell:当前分支:`git branch --show-current`
常用自定义命令集
  • /fix-pr <pr-num> — 读 PR 评论并修复
  • /refactor <file> — 按团队规范重构指定文件
  • /explain <symbol> — 深度解释某个函数/类
  • /migrate <from> <to> — 跨框架迁移
  • /test <file> — 给指定文件补测试
💡 和 Skill 的区别
Slash Command 是"主动触发"的固定 prompt 模板;Skill 是"被动加载"的知识文档。命令适合你经常重复的标准动作,Skill 适合"某类任务时才需要的专业知识"。

23 · Git Worktree · 并行开发工作流

🚀 进阶

Git Worktree 让同一个仓库可以同时 checkout 多个分支到不同目录。Claude Code 原生支持 worktree,让你可以并行跑多个 Claude 实例,互不干扰。

基本用法

# 在隔离的 worktree 中启动一个 Claude
claude --worktree

# 或者让某个 agent 在 worktree 里跑
# (在 Task 工具中指定 isolation: "worktree")

典型场景

  • A/B 实验:让两个 Claude 分别尝试不同方案,比对结果后选一个合并
  • 并行修 bug:一个 Claude 修 bug A,另一个修 bug B,互不冲突
  • 安全试验:在 worktree 跑危险操作,不影响主分支
  • 长任务不阻塞:让 Claude 在 worktree 里跑大重构,主目录继续开发

配置 · 共享 node_modules

每个 worktree 重装一遍依赖太慢。用 symlinkDirectories 让所有 worktree 共用主仓库的 node_modules:

// .claude/settings.json
{
  "worktree": {
    "symlinkDirectories": ["node_modules", ".next/cache", ".venv"],
    "sparsePaths": ["src", "tests", "package.json"]
  }
}
实战 · 让三个 Claude 并行竞标同一个任务
# 终端 1
cd ~/project && claude --worktree
→ "用 React Query 实现数据缓存"

# 终端 2
cd ~/project && claude --worktree
→ "用 Zustand 实现数据缓存"

# 终端 3
cd ~/project && claude --worktree
→ "用 SWR 实现数据缓存"

# 三个并行跑,最后 diff 三个 worktree 的实现,选最好的合并

24 · Output Styles · 定制响应风格

🚀 进阶

Output Style 决定 Claude 回复的"文风"——是话痨式详细解释,还是只给代码不废话,或者像老师一样引导你思考。

内置样式

/output-style               # 查看 / 切换
/output-style default       # 默认
/output-style explanatory   # 详细解释(适合学习)
/output-style learning      # 引导式(Claude 只提问,你动手)

自定义样式

在 ~/.claude/output-styles/<name>.md 创建:

---
name: code-only
description: 只输出代码,不写任何解释
---

你在回答编码问题时,只输出代码本身。
- 不要前言后记
- 不要解释代码做什么
- 不要说"以下是..."或"希望这有帮助"
- 如果必须说明,作为代码里的简短注释
- 多文件修改时用清晰的文件路径头标明

何时用

  • code-only:老手,diff 自己看,不要废话
  • explanatory:学新技术,每一步都想知道为什么
  • learning:教学/面试练习,Claude 不直接给答案
  • strict-review:代码 review 场景,严苛挑毛病不客套
💡 项目级固定样式
在 .claude/settings.json 设置 "outputStyle": "code-only" 可以让整个项目都固定用这个样式,团队统一。

25 · StatusLine · 动态状态栏

🚀 进阶

Claude Code 底部的状态栏可以完全自定义——通过一个 shell 命令生成内容,每次刷新时重新执行。常用来显示当前分支、token 用量、成本、模型等信息。

配置

// ~/.claude/settings.json
{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh",
    "padding": 1
  }
}

StatusLine 脚本示例

#!/bin/bash
# ~/.claude/statusline.sh
# Claude Code 通过 stdin 传入 JSON:
#   { "session_id", "model", "workspace": {"current_dir"}, "cost": {...} }

input=$(cat)
model=$(echo "$input" | jq -r '.model.display_name')
dir=$(echo "$input" | jq -r '.workspace.current_dir' | sed "s|$HOME|~|")
branch=$(cd "$(echo "$input" | jq -r '.workspace.current_dir')" && git branch --show-current 2>/dev/null)
cost=$(echo "$input" | jq -r '.cost.total_cost_usd // 0')

echo "🤖 $model │ 📁 $dir │ 🌿 ${branch:-no-git} │ 💰 \$$(printf '%.3f' $cost)"
高级状态栏能显示
  • 当前模型(Opus/Sonnet/Haiku)+ 是否启用 thinking
  • 本次会话累计 token 和费用
  • git 分支 + 是否有未提交改动
  • 项目名称、CI 状态
  • 当前是否在 worktree 里
  • 后台任务数量

26 · 权限规则深度配置

🚀 进阶

粗粒度的 --dangerously-skip-permissions 不是唯一选择。通过精细的权限规则,你可以既自动化又安全。

规则语法

语法含义
"Read"允许所有 Read 操作
"Bash(npm run test)"精确匹配命令
"Bash(git:*)"前缀通配符:所有 git 子命令
"Edit(src/**)"路径 glob:只能编辑 src 下
"Write(/etc/*)"限制写入范围

三类规则

{
  "permissions": {
    "allow": [
      "Read",
      "Edit(src/**)",
      "Bash(npm:*)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(pytest:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(sudo:*)",
      "Bash(curl:*)",
      "Edit(.env*)",
      "Edit(**/secrets/**)",
      "Write(/etc/**)"
    ],
    "ask": [
      "Bash(git push:*)",
      "Bash(npm publish)"
    ],
    "defaultMode": "acceptEdits"
  }
}

优先级

  1. deny > ask > allow(拒绝优先)
  2. 用户设置 < 项目设置 < 本地设置 < 企业 managed 设置

defaultMode 五种

  • default:每个操作都问
  • acceptEdits:自动接受编辑,危险操作仍问
  • plan:规划模式,只读
  • auto:LLM 分类器自动判断安全性
  • bypassPermissions:完全放行(等同 YOLO)
ℹ️ 企业场景
企业可以把管控规则放在 managed-settings.json,再配合 allowManagedPermissionRulesOnly: true,让用户无法覆盖管理员规则。常见于金融、医疗等需要强合规的环境。

27 · Prompt Caching · 上下文成本优化

🚀 进阶

Claude 支持 Prompt Caching:相同的 prompt 前缀只需计费一次,后续复用可便宜 90%。Claude Code 自动利用这个能力,但你组织 CLAUDE.md 和对话的方式直接影响命中率。

缓存命中的关键

  • 稳定的内容放前面:系统提示 → CLAUDE.md → 工具定义 → 对话历史 → 最新消息
  • 任何改动都会让后续的缓存失效
  • CLAUDE.md 频繁修改 = 每次都重新计费

CLAUDE.md 组织原则

# ✅ 好:稳定部分在前,可变部分在后
## 项目架构 (几乎不变)
## 技术栈 (几乎不变)
## 约定规范 (偶尔调整)
## 当前任务 (经常变 → 放最后)

# ❌ 差:把易变的 TODO 放在文件开头
## 今日 TODO
## 项目架构

/compact 的最佳时机

压缩会打破缓存,不要频繁压缩。策略:

  • 接近上下文上限再压缩(70%+)
  • 任务切换时压缩,比中途压缩更划算
  • 压缩后再工作的第一轮会是"冷启动",成本较高
成本对比(示意)
场景Input tokens 成本
冷启动(无缓存)100%
缓存命中(1 小时内)约 10%
缓存命中 + 增量对话约 10% + 新增部分
💡 实战建议
长会话比多个短会话便宜得多。与其开 10 个新 Claude 处理 10 个小任务,不如用 --continue 在一个会话里依次做完。

29 · CI/CD 集成 · 在 GitHub Actions 里跑 Claude

🚀 进阶

官方提供 anthropics/claude-code-action,可以在 GitHub Actions 里调用 Claude,实现自动 review PR、自动修 issue、定时重构等。

最小示例 · PR 自动 review

# .github/workflows/claude-review.yml
name: Claude Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            审查这个 PR 的 diff,重点关注:
            1. 安全漏洞(SQL 注入、XSS、硬编码 secret)
            2. 性能问题(N+1 查询、同步阻塞)
            3. 测试覆盖是否充分
            以 GitHub PR review 的形式输出,有问题的地方用 inline comment。

Issue 触发修 bug

on:
  issue_comment:
    types: [created]

jobs:
  fix:
    if: contains(github.event.comment.body, '/claude fix')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            Issue #${{ github.event.issue.number }} 标题:${{ github.event.issue.title }}
            正文:${{ github.event.issue.body }}

            分析这个 bug,修复后创建一个 PR。
进阶玩法
  • 每晚跑 Claude 清理 TODO 注释、升级依赖
  • 监听特定标签的 issue 自动创建修复 PR
  • 定期生成项目健康报告、代码质量趋势
  • 文档和代码漂移检测,自动补齐

30 · Effort Level 与 Thinking 深度控制

🚀 进阶

Claude 4.5/4.6 支持三档 effort level,直接影响"思考深度"和 token 成本。用对地方能省很多钱。

三档设置

Effort特点适用
low快速响应,最少思考简单问题、格式化、文档
medium平衡(默认)日常编码、普通重构
high深度思考,高成本架构设计、难题 debug、安全审查

切换方式

# 会话中临时切换
/effort high
/effort low

# 持久化
// ~/.claude/settings.json
{
  "effortLevel": "high",
  "alwaysThinkingEnabled": true
}

实战策略

  • 混用策略:日常 medium,遇到难题临时 /effort high,搞定后回 medium
  • SubAgent 分工:给 code-reviewer、architect 这类 agent 配 high,给 formatter、doc-writer 配 low
  • 模型 × effort 组合:Haiku + low 干粗活、Sonnet + medium 日常、Opus + high 攻坚
💡 何时开 thinking
当任务涉及"多步推理、多方案权衡、微妙 bug"时开 thinking。纯写代码、改格式、查 API 文档关掉反而更快更便宜。用 /fast 快速切换 fast mode。

31 · 多实例并行 · tmux / 分屏工作流

🚀 进阶

单个 Claude 是顺序的,但你的大脑和项目常常是并行的。用 tmux(或 iTerm2 的分屏、zellij)同时开多个 Claude 实例,能成倍提升吞吐。

典型布局

┌─────────────────┬─────────────────┐
│  Pane 1         │  Pane 2         │
│  主开发 Claude   │  测试/文档       │
│  (Opus, plan)   │  (Haiku, 快速)   │
├─────────────────┼─────────────────┤
│  Pane 3         │  Pane 4         │
│  日志监控        │  后端 dev server │
│  (Claude -p)    │  (纯 shell)      │
└─────────────────┴─────────────────┘

实战配方

  1. Pane 1:主开发 Claude,负责核心逻辑,启动时用 --worktree 隔离
  2. Pane 2:专用测试 agent,claude -c 接着写测试或跑测试,用 Haiku 省钱
  3. Pane 3:把错误日志 tail 给另一个 Claude tail -f app.log | claude -p "实时盯着有没有异常"
  4. Pane 4:普通 shell,跑服务、数据库、调试工具

tmux 快速脚本

#!/bin/bash
# dev-session.sh - 一键打开 4 窗格开发环境
tmux new-session -d -s dev -c ~/myproject
tmux split-window -h -c ~/myproject
tmux split-window -v -c ~/myproject
tmux select-pane -t 0
tmux split-window -v -c ~/myproject

tmux send-keys -t 0 'claude --worktree' Enter
tmux send-keys -t 1 'claude -c' Enter
tmux send-keys -t 2 'npm run dev' Enter
tmux send-keys -t 3 'tail -f logs/app.log' Enter

tmux attach -t dev
⚠️ 注意会话冲突
多个 Claude 同时改同一批文件会互相踩脚。要么用 worktree 物理隔离,要么明确分工(A 只改前端 / B 只改后端)。

32 · 上下文工程 · Context Engineering

🚀 进阶

"Prompt 工程"关注单次提问,"上下文工程"关注如何在整个会话周期管理信息流。这是高手和新手最大的差距之一。

六个核心原则

① 信息分层:短期 vs 长期

  • 长期(CLAUDE.md、Skill、系统提示)— 稳定、被缓存、跨会话
  • 中期(当前会话)— 本次任务相关
  • 短期(最新几轮)— 当前具体动作

别把"本次任务细节"塞进 CLAUDE.md;别指望 Claude 长期记住会话里说过的话。

② 主动引用而非被动等待

与其问"你能读 config.py 吗",不如直接 @config.py 把文件显式引入上下文。@ 前缀能引用任何文件/目录/URL。

③ 及时清理噪音

一轮失败的尝试、一堆无关的 grep 输出、调试用的 print—— 这些都会污染上下文。/compact 或直接 /clear 重新开始,比让 Claude 在垃圾里找信号强。

④ 锚定关键决策

重大决策(架构选型、API 合约、命名规范)一旦确定,立即让 Claude 写进 CLAUDE.md,防止后续对话漂移。

⑤ 显式传递文件依赖

让 Claude 在修改前先 Read 相关的所有上下游文件。好提示:

你:修改 user_service.py 前,先读 user_model.py、user_routes.py、
tests/test_user.py,理解调用关系后再动手。

⑥ 结构化回顾

长任务每隔一段时间让 Claude 总结:

你:总结一下我们到目前为止做了哪些决定、改了哪些文件、还剩什么

这个总结本身就是下一段工作的"压缩上下文",比 /compact 更受控。

对比:普通 vs 高阶

普通用法:一口气说完需求 → Claude 到处乱读 → 上下文爆炸 → 回答质量下降 → 开新会话从头来

高阶用法:

  1. 先 /init 写好 CLAUDE.md 锚定架构
  2. 进入 Plan Mode 让 Claude 先理解,不动手
  3. 批准计划后退出 Plan,按计划执行
  4. 每完成一个阶段让 Claude 总结 + 写进 CLAUDE.md
  5. 任务切换前 /compact 或 /clear
💡 心法
把 Claude 当成一个"有 200k token 短期记忆的天才实习生"。他很聪明,但会忘,容易被无关信息带偏。你的工作是当好"信息过滤器"和"结构化笔记本",让他始终聚焦在真正重要的东西上。
