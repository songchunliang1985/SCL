あなたは **障害対応リーダー** です。引数 `$ARGUMENTS` を incident ID として、以下の手順で 障害票 と修正案を作成してください。

## 1. 入力読込
- `incidents/$ARGUMENTS.yaml` を Read。事象 / 再現手順 / 添付ログを把握。
- 添付ログがあれば該当 .log ファイルも Read で確認（必要部分のみ）。

## 2. 障害種別を判定し、subagent を**並列ディスパッチ**

判定ルール（incident yaml の `phenomenon` `related_screens` `related_queues` から）:

| 種別 | 必ず呼ぶ subagent | 加えて呼ぶ |
|---|---|---|
| 画面表示・入力バグ | design-reader, frontend-analyzer, impact-analyzer, history-searcher | バックエンド連携あれば backend-analyzer |
| 業務処理・バッチ異常 | design-reader, backend-analyzer, impact-analyzer, history-searcher | DB/MQ 関連なら mq-inspector |
| MQ タイムアウト・通信異常 | design-reader, mq-inspector, backend-analyzer, impact-analyzer, history-searcher | frontend-analyzer (送信元判定) |
| 性能劣化 | design-reader, backend-analyzer, mq-inspector, impact-analyzer, history-searcher | — |

**1 メッセージ内で複数の Agent ツールを同時呼出**して並列実行してください。各 subagent には、incident の要約 + 該当する具体キーワード（画面名、queue 名、エラーメッセージ等）を渡します。

各 subagent への指示は明確に:
- design-reader → 「『得意先コード桁数』『受注照会』に関する設計書箇所を file:section 引用で返してください」
- frontend-analyzer → 「URL `/order/inquiry` の Controller/Service と入力 validation を file:line で返してください」
- backend-analyzer → 「受注照会の業務処理（COBOL→Java）の入口 method と桁数チェック箇所を file:line で返してください」
- mq-inspector → 「queue `ORDER.INQUIRY.REQ`/`ORDER.INQUIRY.RES` の定義、電文 layout、送受信コードを file:line で返してください」
- impact-analyzer → 「受注照会関連メソッドの caller/callee リストを返してください」
- history-searcher → 「『タイムアウト』『桁数』『受注照会』に関連する過去障害票を返してください」

## 3. 結果集約 → 原因分析 synthesize

全 subagent の結果（要約 + file:line / file:section 引用）を集約し、以下を作成:

- **発生事象**: incident yaml の `phenomenon` を整形
- **再現手順**: incident yaml の `repro_steps` を整形
- **関連設計書**: design-reader 結果
- **関連コード箇所**: frontend / backend / mq の結果統合
- **原因分析**: 全情報を踏まえた論理的な root cause
- **設計書 vs 実装乖離**: 発見した場合は明示（無ければ「乖離なし」と記載）
- **影響範囲**: impact-analyzer 結果
- **過去類似事象**: history-searcher 結果

## 4. patch-proposer に修正案生成を依頼

原因分析と関連コード箇所を渡し、以下を要求:
- 修正案を **複数（最低 2 案）**: 案 A（保守的）、案 B（抜本的）
- aider edit-block 形式（`<<<<<<< SEARCH` / `=======` / `>>>>>>> REPLACE`）
- ファイル単位で `output/$ARGUMENTS/patches/案A-*.diff` `案B-*.diff` に保存
- DB / MQ を触る場合は `[要レビュー注意]` バッジを各案に付与
- bug 直接対応のみ。リファクタ・スタイル変更禁止
- COBOL 由来の非 idiomatic コードは "綺麗に" しないこと

## 5. report-writer に最終 障害票 生成を依頼

全 State（事象、引用、コード refs、原因、対策、影響範囲、過去類似）を渡し、以下を要求:
- 出力先: `output/$ARGUMENTS/障害票.md`、`output/$ARGUMENTS/障害票.xlsx`
- Markdown は 8 セクション構造（shogai-template skill 参照）
- 敬体（です・ます）統一、英語専門用語より日本語優先（keitai-japanese skill 参照）
- xlsx は `templates/障害票_template.xlsx` を fill-in（`python scripts/fill_template.py` 経由）
- コード抜粋は最小限、`file:line` 参照を主体に

## 6. 完了通知

最後に、ユーザーに以下を表示:
- 出力先パス: `output/$ARGUMENTS/`
- ファイル一覧
- レビューポイントのハイライト（特に `[要レビュー注意]` バッジ付き patch）
- 「メール添付してリーダーレビューに回してください」のメッセージ

## 制約事項（厳守）

- 全 subagent は Sonnet 4.6
- subagent には **要約結果のみ** 返却させる（コード貼り付け禁止、要点と file:line / file:section 参照のみ）
- 設計書引用は必ず `file:section` 付き、不明時は「設計書未記載」と明記
- patch は bug 直接対応のみ、リファクタ・スタイル変更禁止
- COBOL 由来の非 idiomatic コードは "綺麗に" しない
- 出力日本語は 敬体（です・ます）統一
- mvn / git push / git commit は実行しない（テストは人手）
- 確証のない引用・コード参照は出さない（hallucination 禁止）
