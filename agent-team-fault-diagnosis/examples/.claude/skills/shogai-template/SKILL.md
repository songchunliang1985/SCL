# 障害票テンプレート構造

社内 障害票 は以下の 8 セクション構造で統一します。**全セクション必須**、情報なしの場合も「該当なし」と明記（空欄禁止）。

---

## 全体構造

```markdown
# 障害票 INC-YYYYMM-NNN

## 1. 発生事象
## 2. 再現手順
## 3. 関連設計書
## 4. 関連コード箇所
## 5. 原因分析
   ### 5.1 設計書 vs 実装乖離
## 6. 影響範囲
## 7. 対策案
   ### 7.1 案 A
   ### 7.2 案 B
## 8. 推奨案と次アクション
```

---

## 1. 発生事象

**目的**: 何が起きているかを簡潔に伝える。

**記入項目**:
- 日時（YYYY-MM-DD HH:MM）
- 報告者
- 重要度（高 / 中 / 低）
- 業務影響（影響業務、影響人数、停止時間等）
- 事象記述（敬体、3-5 行程度）

**記入例**:
```markdown
## 1. 発生事象

- 日時: 2026-04-15 10:23
- 報告者: 山田太郎
- 重要度: 高
- 業務影響: 営業部門 30 名が照会業務不可

受注照会画面（URL: /order/inquiry）にて、得意先コード欄に 10 桁を入力した場合、
約 60 秒後にタイムアウトエラーが発生いたします。9 桁以下の入力では正常動作を確認しております。
```

---

## 2. 再現手順

**目的**: 第三者が同じ事象を再現できる手順。

**記入項目**:
- 番号付き箇条書き
- 必要な前提条件（テストデータ、環境）
- 期待結果と実際結果の差分

**記入例**:
```markdown
## 2. 再現手順

前提: 検証環境 (https://test.example.com)、テスト得意先コード `1234567890` (10桁) を使用

1. 受注照会画面を開きます。
2. 得意先コード欄に `1234567890` を入力します。
3. 「照会」ボタンを押下します。
4. 60 秒後、`java.net.SocketTimeoutException` が発生いたします。

期待結果: 正常に検索結果が表示される、または「桁数エラー」表示
実際結果: タイムアウトエラー
```

---

## 3. 関連設計書

**目的**: 仕様の根拠を引用付きで明示。

**記入項目**:
- 設計書ファイル名 + section 番号
- 抜粋（200 字以内）
- design-reader subagent の結果をそのまま引用

**記入例**:
```markdown
## 3. 関連設計書

### 3.1 受注機能_詳細設計書 第 2 章 受注照会
- 出典: `docs-md/受注機能_詳細設計書/03-受注照会.md:section-2.3`
- 元ファイル: 受注機能_詳細設計書.docx (page 18)
- 抜粋:
  > 得意先コードは 9 桁固定とする。10 桁以上の入力は受け付けない仕様。

### 3.2 共通入力規約 第 5 節
（同様）
```

該当設計書がない場合:
```markdown
## 3. 関連設計書

該当設計書記載なし（設計書側に該当仕様の記述がございません）。
```

## 4. 関連コード箇所

**目的**: 障害に関わるコードの所在を `file:line` で明示。

**記入項目**:
- frontend / backend / mq の各 subagent 結果統合
- ファイル / 行番号 / 概要
- コード抜粋は最小限（行番号で十分）

**記入例**:
```markdown
## 4. 関連コード箇所

### 4.1 Frontend
- `src/main/java/com/example/order/OrderInquiryController.java:45` リクエスト受付
- `src/main/java/com/example/order/OrderInquiryForm.java:12` 入力検証 (`@Pattern(regexp="\\d{1,9}")`)

### 4.2 Backend
- `src/main/java/com/example/order/OrderInquiryService.java:88` 業務処理入口
- `src/main/java/com/example/order/OrderInquiryDao.java:60` DB アクセス（JDBC timeout 60s）

### 4.3 MQ
- `config/mq/order.mqsc:line 10` `ORDER.INQUIRY.REQ` queue 定義
- `src/main/java/com/example/order/copybook/OrderInquiryReqMessage.java:5` 電文 layout
```

## 5. 原因分析

**目的**: 根本原因の論理的説明。

**記入項目**:
- 観察された事実
- 仮説と検証
- 根本原因
- 設計書との比較（5.1）

**記入例**:
```markdown
## 5. 原因分析

### 観察された事実
- Form 側の検証は 9 桁まで許容しております。
- Service 側では桁数チェックを行わず DAO へ渡しております。
- DAO の SQL は `WHERE customer_cd = ?` で、固定長 10 桁を前提としております。

### 仮説と検証
- 仮説: クライアント側 JS の検証を bypass された 10 桁入力が、Service / DAO まで到達している。
- 検証: ログ確認により、当該リクエストが Controller に到達していることを確認いたしました。

### 根本原因
DAO の SQL が固定長 10 桁を前提としているため、10 桁入力時は filler が無くなり、
インデックスが効かない全件 scan に近い挙動となり、JDBC timeout (60 秒) に到達いたします。

### 5.1 設計書 vs 実装乖離

**乖離あり**

- 設計書: 「得意先コードは 9 桁固定、10 桁以上は受け付けない仕様」
- 実装: Form は 9 桁まで許容、Service / DAO は 10 桁を許容
- 評価: Service / DAO 側の桁数チェック漏れと判断いたします。
```

## 6. 影響範囲

**目的**: デグレ確認用に変更影響を網羅。

**記入項目**:
- impact-analyzer subagent 結果
- caller / callee 一覧
- 影響レベル（低 / 中 / 高）
- デグレ確認推奨ポイント

**記入例**:
```markdown
## 6. 影響範囲

### Caller（呼出元）
| ファイル | 行 | 影響レベル |
|---|---|---|
| `OrderInquiryController.java` | 50 | 低 |
| `OrderInquiryBatchService.java` | 120 | 中 |

### Callee（内部呼出）
- `OrderInquiryDao.findByCustomerCd`
- `OrderValidator.validateCustomerCd`

### 総合評価
**中** — 受注照会画面 + 夜間バッチが影響、両方の動作確認が必要です。

### デグレ確認推奨
- 受注照会画面（9桁・8桁・1桁）
- 夜間バッチ（過去 1 週間分のデータ）
- 単体テスト `OrderInquiryServiceTest`
```

---

## 7. 対策案

**目的**: 修正方針を複数案提示。

**記入項目**:
- 最低 2 案（保守的・抜本的）
- 各案の概要、修正範囲、リスク、レビュー観点
- patch ファイルへの参照

**記入例**:
```markdown
## 7. 対策案

### 7.1 案 A（保守的）

- **概要**: Form の `@Pattern` を `\d{9}` に変更し、ちょうど 9 桁のみ受付ます。
- **修正範囲**: 1 ファイル, 1 行
- **patch**: `patches/案A-form-validation.diff`
- **リスク**: 既に 8 桁以下を運用している顧客があれば影響あり。
- **レビュー観点**: 通常レビュー

### 7.2 案 B（抜本的）

- **概要**: Form / Service / DAO の三層で桁数を防御的に検証いたします（defense in depth）。
- **修正範囲**: 3 ファイル, 6 行
- **patch**: `patches/案B-defense-in-depth.diff`
- **リスク**: 結合テスト範囲が広くなります。
- **レビュー観点**: [要レビュー注意] DB アクセス変更を含むため、結合テスト必須

```

---

## 8. 推奨案と次アクション

**目的**: 担当者・リーダーが判断・実行できる形にまとめる。

**記入項目**:
- 推奨案
- 推奨理由
- 次アクション（担当割当、テスト、リリース計画）
- 過去類似事象への参照

**記入例**:
```markdown
## 8. 推奨案と次アクション

### 推奨案
**案 B（抜本的）** を推奨いたします。

### 推奨理由
- 過去事例 INC-202312-007 にて同種の事象が発生しており、Form/Service 二重チェック方針が有効であった実績がございます。
- 設計書遵守の観点からも、Service / DAO 側に防御層を持つことが望ましいと判断いたします。

### 次アクション
1. 担当者割当: 受注機能担当 田中 様
2. 単体テスト: 修正各層のテストケース追加
3. 結合テスト: 受注照会画面 + 夜間バッチ
4. リリース計画: 次回月次リリース (2026-04-30) を想定

### 関連 過去障害票
- `past-incidents/INC-202312-007.md` 顧客検索画面の同種事象
- `past-incidents/INC-202309-015.md` COMP-3 桁あふれ事例

### レビュー観点
- [要レビュー注意] DB アクセス変更を含む
- copybook の固定長前提は変更しない方針
```
